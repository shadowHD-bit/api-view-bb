Partner Client API

Описание внешних методов для партнёрской интеграции: идентификация клиента, создание / обновление клиента партнёра, получение лимитов, курс обмена, создание СБП-инвойса, получение статуса и webhook

Общее описание

Способ идентификации клиента определяется для партнёра на этапе заключения договора и сохраняется в настройках партнёра. От выбранного способа зависит, какие данные передаются партнёром и где выполняется идентификация клиента при создании СБП-инвойса

BY_EXTERNAL_DATA

Партнёр передаёт данные клиента через POST /api/v1/partner-clients. Система создаёт или обновляет клиента партнёра и выполняет необходимые проверки

данные от партнёра

BY_EXTERNAL_ID

Партнёр передаёт только внешний идентификатор клиента. Система использует его для поиска или создания клиента партнёра, а KYC выполняется на стороне платформы

только external_id

BY_BB_LOGIN

Партнёр передаёт внешний идентификатор как reference. Клиент определяется на стороне платформы после перехода на страницу инвойса

идентификация платформой
Ключевая идея: partner_client_external_id передаётся в инвойсе всегда, но его смысл зависит от типа идентификации. Для BY_BB_LOGIN он не используется для поиска клиента партнёра и хранится как технический reference
Если в инвойсе не используется обменная операция, блок exchange_info не возвращается в ответах и webhook

Подпись запросов

Для внешних методов используется HMAC-подпись. Конкретная строка для подписи зависит от метода

Общие правила

  • Партнёр передаёт публичный API-ключ в заголовке X-API-KEY
  • Поле sign передаётся в теле запроса
  • Подпись рассчитывается на стороне партнёра и проверяется на стороне платформы
  • Если подпись не совпадает, запрос отклоняется

Подпись для создания / обновления клиента партнёра

Для POST /api/v1/partner-clients подпись рассчитывается от канонического JSON тела запроса без поля sign

sign = hmac_sha256(canonical_json(body_without_sign), api_key.secret)

Пример тела, которое подписывается

{
  "client_id": "client-001",
  "last_name": "Иванов",
  "first_name": "Иван",
  "phone": "+79990001122",
  "birth_date": "1990-01-01",
  "passport": "4510123456",
  "passport_issue_date": "2010-01-15"
}

Пример запроса с подписью

{
  "client_id": "client-001",
  "last_name": "Иванов",
  "first_name": "Иван",
  "phone": "+79990001122",
  "birth_date": "1990-01-01",
  "passport": "4510123456",
  "passport_issue_date": "2010-01-15",
  "sign": "<calculated_sign>"
}

Подпись для создания инвойса

Для POST /api/v1/invoices подпись рассчитывается по строке из полей currency, amount, header, description

sign = hmac_sha256(currency + amount + header + description, api_key)

Для строки

RUBR10000ООО РомашкаОплата заказа #A-1024

подпись будет передана в поле

{
  "sign": "<calculated_sign>"
}

Подписи в ответах и webhook

В ответах по инвойсу и webhook могут возвращаться две подписи

ПолеОписание
signПодпись ответа, рассчитанная с использованием API key
sign_2Подпись ответа, рассчитанная с использованием API secret
Формат канонического JSON и точный секрет для HMAC должны быть едиными для партнёра и платформы, иначе проверка подписи будет невозможна

Webhook по изменению статуса инвойса

Платформа отправляет webhook на URL партнёра при изменении статуса инвойса. Payload соответствует формату GET /api/v1/invoices

Webhook содержит sign и sign_2 для проверки подлинности payload

Пример payload

{
  "sign": "<calculated_sign_by_api_key>",
  "sign_2": "<calculated_sign_by_api_secret>",
  "payed": true,
  "id": "1izt8r3YNoZ6kgwewB3xCB",
  "amount": 990,
  "currency": "RUBR",
  "payed_amount": 990,
  "partner_client_external_id": "client-001",
  "partner_client_id": "pc_123456",
  "partner_client_identification_type": "BY_BB_LOGIN",
  "data": {
    "order_id": "BUY-1024"
  },
  "sbp_info": {
    "amount": 1000,
    "currency_symbol": "RUBR",
    "status": "captured",
    "code": "0"
  },
  "exchange_info": {
    "from_currency": "RUBR",
    "from_amount": 990,
    "to_currency": "USDT",
    "network": "TRC20",
    "actual_rate": 99.8,
    "actual_to_amount": 9.91983967,
    "exchange_status": "completed",
    "exchanged_at": "2026-04-30T12:03:15Z"
  }
}
Если для инвойса не используется обменная операция, блок exchange_info не возвращается в webhook

Правила обработки

  • Webhook считается доставленным, если партнёр вернул HTTP 2xx
  • Если webhook не принят, используется текущая retry-логика
  • Партнёр связывает webhook со своей заявкой через data.order_id
  • Партнёр определяет результат по payed и sbp_info.status
